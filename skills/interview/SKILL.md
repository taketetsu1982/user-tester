# /interview

ターゲットユーザーから1人のペルソナを生成し、ユーザー（人間）がそのペルソナに自由に質問できる対話セッションを開始する。

## Input

以下のドキュメントが存在すれば読み込む。あるほどペルソナの精度が上がる。

- PRDドキュメント — ターゲットユーザー、ペイン、提供価値
- MRDドキュメント — 市場背景、競合、ターゲットセグメント
- ユーザーストーリー — ストーリー（存在すれば）
- UI仕様 — 画面定義（存在すれば）

> **注:** 入力ファイルのパスはプロジェクトごとに異なる。導入時にプロジェクトの規約に合わせること。
> いずれのドキュメントも存在しない場合は、Step 1 でユーザーにヒアリングして情報を収集する。

## Output

インタビューサマリをユーザーに提示する（ファイル出力はしない）。

## Execution Steps

### Step 1: コンテキストを収集する

PRD・MRD等のドキュメントがあれば読み込む。

**ドキュメントがない場合**、ユーザーに以下をヒアリングする:

1. **プロダクト概要**: 何をするプロダクトか（一言で）
2. **ターゲットユーザー**: 誰が使うか（ロール・年齢層・職業など）
3. **ユーザーのペイン**: ターゲットが抱えている課題
4. **提供価値**: プロダクトがどう解決するか
5. **主な機能**: 画面や操作フロー（わかる範囲で）

ヒアリング結果を以降のステップでPRDの代わりに使用する。

### Step 2: ペルソナを選択する

ユーザーに以下を確認する:

1. **ロール**: ターゲットのどのロールか
2. **属性の方向性**: 以下から選択またはカスタム指定
   - 典型的なユーザー
   - ITリテラシーが低いユーザー
   - ヘビーユーザー
   - エクストリームユーザー（具体的な軸を指定）
   - カスタム（自由に属性を指定）

指定がなければ、メインターゲットに最も近い典型的なペルソナを1人生成する。

ペルソナ定義（基本項目）:

| 項目 | 内容 |
|---|---|
| 名前 | 架空の日本語名 |
| 年齢・職業 | ターゲットユーザー像に基づく |
| ITリテラシー | 低 / 中 / 高 |
| 利用シーン | 主な利用シーンから具体化 |
| 抱えている課題 | ユーザーのペインから具体化 |
| 性格・行動傾向 | 慎重派 / せっかち / 探索好き など |
| このプロダクトへの期待 | 提供価値に対する具体的な期待 |
| エクストリーム属性 | 該当する場合のみ |

競合製品、日常ツール、業界特有の属性等の情報があれば、対話の深みを出すために追加項目としてペルソナに反映する。

### Step 3: interviewee エージェントを起動する

Agent ツールで interviewee エージェントを指定し、name を付けて起動する。
prompt に以下を含める:
- ペルソナ定義（Step 2 で生成したもの）
- プロダクト概要（提供価値・機能概要を要約）
- 画面・操作フローの要約（UI仕様等があれば）
- 最初の質問（「まず自己紹介をお願いします」等、対話の起点となるメッセージ）

エージェントの回答をユーザーに提示し、以下を伝える:

```
{ペルソナ名}（{年齢}・{職業}）とのインタビューを開始しました。
自由に質問してください。終了するときは「インタビュー終了」と入力してください。
```

### Step 4: 対話を中継する

ユーザーの質問を受け取ったら、Agent ツールの SendMessage で interviewee エージェントに転送する。
エージェントの回答をユーザーにそのまま提示する。
「インタビュー終了」が入力されるまでこのサイクルを繰り返す。

### Step 5: インタビューサマリを作成する

対話終了後、やり取りの内容を以下を含むサマリとして要約する:

- **主要な発見**（ペルソナが強く反応したポイント）
- **ペルソナの態度**（プロダクトへの関心・課題の深刻度・乗り換え意欲を、根拠となる発言とともに）
- **示唆**（企画・設計への示唆）

## 注意

- ペルソナはターゲットユーザー像から逸脱しない
- 把握していない機能については「知らない」と答えるのが自然
- インタビュアー（ユーザー）の質問を誘導しない。聞かれたことに答える
- 同じペルソナで複数回インタビューする場合、前回の文脈は引き継がない（毎回新規）
