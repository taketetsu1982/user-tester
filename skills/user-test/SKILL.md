# /user-test

ターゲットユーザーからペルソナを生成し、Chrome連携でブラウザを実際に操作してユーザビリティのフィードバックを収集する。
ペルソナは1人ずつテストし、次に進むかはユーザーが判断する。

## 前提確認

1つでもNoならユーザーに案内して中断する。

- Claude Code が Chrome 連携済みであること（`--chrome` または `/chrome`）
- テスト対象のアプリが起動中でURLにアクセス可能であること

## Input

以下のドキュメントが存在すれば読み込む。あるほどペルソナとシナリオの精度が上がる。

- PRDドキュメント — ターゲットユーザー、ペイン、提供価値
- MRDドキュメント — 市場背景、競合、ターゲットセグメント
- ユーザーストーリー — ストーリーとシナリオ（存在すれば）
- UI仕様 — 画面定義・インタラクション（存在すれば）

> **注:** 入力ファイルのパスはプロジェクトごとに異なる。導入時にプロジェクトの規約に合わせること。
> いずれのドキュメントも存在しない場合は、Step 1 でユーザーにヒアリングして情報を収集する。

## Output

テスト結果をユーザーに提示する（ファイル出力はしない）。

## Execution Steps

### Step 1: コンテキストを収集する

PRD・MRD等のドキュメントがあれば読み込む。

**ドキュメントがない場合**、ユーザーに以下をヒアリングする:

1. **プロダクト概要**: 何をするプロダクトか（一言で）
2. **ターゲットユーザー**: 誰が使うか（ロール・年齢層・職業など）
3. **ユーザーのペイン**: ターゲットが抱えている課題
4. **提供価値**: プロダクトがどう解決するか
5. **主な機能**: 画面や操作フロー（わかる範囲で）

ヒアリング結果を以降のステップでPRDの代わりに使用する。

ユーザーにテスト対象のURLを確認する。

### Step 2: ペルソナを生成する

ターゲットユーザー情報からテスト用ペルソナを生成する。
ペルソナの定義項目はプロダクトの内容に応じて決める。以下は基本項目:

| 項目 | 内容 |
|---|---|
| 名前 | 架空の日本語名 |
| 年齢・職業 | ターゲットユーザー像に基づく |
| ITリテラシー | 低 / 中 / 高 |
| 利用シーン | 主な利用シーンから具体化 |
| 抱えている課題 | ユーザーのペインから具体化 |
| 性格・行動傾向 | 慎重派 / せっかち / 探索好き など |
| このプロダクトへの期待 | 提供価値に対する具体的な期待 |
| エクストリーム属性 | 該当する場合のみ。極端な特性とその背景 |

競合製品、日常ツール、ドメイン固有の属性等の情報があれば、追加項目としてペルソナに反映する。

**生成ルール:**
- ターゲットのロールごとに最低1人生成する
- ITリテラシーと性格のバリエーションを持たせる（全員同じにしない）
- 一般ユーザーと管理者など複数ロールがある場合、両方のペルソナを生成する
- **エクストリームユーザーを必ず含める。** 利用頻度・スキル・身体的制約・環境・動機・データ量などの軸から、プロダクトの文脈に合う極端な特性を持つユーザーを最低1人生成する

ペルソナ一覧をユーザーに提示し、調整の指示があれば反映する。
テスト順序もこのタイミングでユーザーに確認する。

### Step 3: テストシナリオを組み立てる

ユーザーストーリーのシナリオ（Given/When/Then）をベースに、ペルソナが実際に体験する操作手順を構成する。

各シナリオについて:
1. **対象ストーリー**: US-ID を紐付ける
2. **操作の流れ**: 画面遷移に沿った具体的な操作手順（URLからの遷移を含む）
3. **注目ポイント**: 特に検証したい観点（初見での発見しやすさ、操作の手数、エラー時の挙動等）

### Step 4: ペルソナ1人ずつテストする

**1人ずつ順番に** テストする。各ペルソナのテスト完了後、ユーザーに次のペルソナに進むか確認する。

各ペルソナについて:
1. このペルソナの人物像をユーザーに提示する
2. テストシナリオの操作手順に沿って Chrome でブラウザを操作する:
   - 対象URLに遷移する
   - シナリオの各ステップを実際に操作する（クリック、入力、遷移）
   - 各画面でペルソナの視点から以下を評価する:
     - **第一印象**: 画面を見た瞬間、このペルソナは何に目がいくか
     - **迷いポイント**: 次に何をすべきかわかるか。ラベルやボタンの意味が伝わるか
     - **操作の手数**: 目的達成までのステップ数は適切か
     - **エラー体験**: バリデーションエラーやエッジケースの挙動
   - コンソールログにエラーがあれば記録する
3. このペルソナのフィードバックをまとめてユーザーに提示する
4. **ユーザーに確認する**: 「次のペルソナ（{名前}）に進みますか？ テストを終了しますか？」

ユーザーが終了を選択したら Step 5 へ進む。

### Step 5: フィードバックを統合する

テスト済みの全ペルソナのフィードバックを統合し、以下を含むサマリを作成する:

- **ペルソナ別フィードバック**（第一印象、操作のわかりやすさ、目的達成のしやすさ、つまずき、改善提案）
- **共通の発見**（強み・課題）
- **技術的な発見**（コンソールエラー、パフォーマンス、レスポンシブ）
- **改善提案**（優先度順。どのペルソナがどう困ったかを根拠として付記）

## 注意

- ペルソナはターゲットユーザー像から逸脱しない。架空の属性を過度に追加しない
- 把握している画面・機能の範囲でテストする
- フィードバックは「ペルソナとしての感想」であり、実装指示ではない。改善の判断はユーザーに委ねる
- データの書き込み（登録・削除等）を伴う操作がある。テスト用データで実施すること
- ログインが必要な場合、CAPTCHA等はユーザーに手動対応を依頼する
