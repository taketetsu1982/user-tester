# /user-survey

PRDのターゲットユーザーから多数のペルソナを生成し、設問に回答させて定量データを収集・統計サマリを作成する。

## Input（すべて読み込む）

- PRDドキュメント — ターゲットユーザー、ペイン、提供価値
- product-model.json — entities, actors, screens, transitions
- ユーザーストーリー — ストーリー（存在すれば）
- UI仕様 — 画面定義（存在すれば）

> **注:** 入力ファイルのパスはプロジェクトごとに異なる。導入時にプロジェクトの規約に合わせること。

## Output

アンケート結果の統計サマリをユーザーに提示する（ファイル出力はしない）。

## Execution Steps

### Step 1: 企画書を読み込む

PRDとproduct-model.jsonは必須。存在しないかテンプレート状態の場合は中断する。

### Step 2: 設問を設計する

調査目的に応じた設問セットを作成する。ユーザーに調査したいテーマを確認する。

**設問の構成（目安 5〜10問）:**
- 7段階評価（リッカート尺度）: 3〜5問。「この機能に魅力を感じるか」「使いたいと思うか」等
- 選択式: 1〜3問。「最も重要な機能はどれか」「どの場面で使いたいか」等
- 自由記述: 1問。「率直な感想・懸念」

設問をユーザーに提示し、承認を得てから次へ進む。

### Step 3: 回答者の属性分布を定義する

PRDのターゲットUser像をベースに、回答者の属性分布を設計する。

| 属性 | 分布例 |
|---|---|
| ロール | actors の比率に応じて配分 |
| ITリテラシー | 低20% / 中50% / 高30%（PRDに応じて調整） |
| 利用頻度想定 | ヘビー20% / 通常60% / ライト20% |
| 年齢層 | ターゲットの中心±幅を持たせる |

**エクストリームユーザーを含める:**
- 全回答者の10〜15%をエクストリーム属性にする
- 利用頻度・スキル・環境・動機の極端なバリエーションを混ぜる

回答者数はユーザーに確認する（デフォルト: 100人）。

### Step 4: survey-respondent エージェントを並列起動する

回答者を10〜20人単位のバッチに分け、survey-respondent エージェントを並列起動する。

Agent ツールで survey-respondent エージェントを指定し、prompt に以下を含める:
- バッチ内の回答者属性リスト（名前は不要。属性の組み合わせのみ）
- 設問セット
- プロダクト概要（PRDの提供価値・機能概要を要約）
- 画面・操作フローの要約（UI仕様または product-model.json の screens から）

### Step 5: 回答を集計・統計サマリを作成する

全バッチの回答を統合し、以下を含むサマリを作成する:

- **回答者属性の内訳**（ロール・ITリテラシー・利用頻度の分布）
- **設問別集計**（7段階評価: 100点スケールに変換した平均・中央値・属性別クロス集計、選択式: 選択肢ごとの割合、自由記述: 頻出キーワードと代表回答）
  - 100点変換: `(回答値 - 1) / 6 × 100`（1→0点、4→50点、7→100点）
- **総括**（強いシグナル、注意シグナル、/user-test・/user-interview で深掘りすべきテーマ）

## 注意

- 設問は調査目的に集中させる。関係ない設問を入れない
- 回答のバラつきはペルソナ属性の違いから自然に生まれるべき。意図的に偏らせない
- 統計サマリは事実ベースで記述する。過度な解釈を加えない
- 「深掘り推奨」で /user-test・/user-interview への接続を示す
- survey-respondent エージェントは haiku モデルを使用する。定型的な設問回答を大量に生成するタスクであり、コスト効率を優先した選択
